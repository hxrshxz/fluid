# Fluid Weekly Automation Workflow

This plan defines a lightweight queue workflow for weekly PR planning and execution using only Python 3 stdlib and local repo files.

## Goal

Create deterministic scripts that:

- parse issue #5676 checklist items by week,
- chunk work into package-wave units,
- track machine state across runs,
- generate execution prompts for `fluid-pr-agent`,
- report queue status and throughput.

## Files

- `automation/weekly_queue.yaml`: human-readable queue generated by `scripts/plan_week.py`.
- `automation/state.json`: machine state with chunk status, notes, and PR URL tracking.
- `scripts/plan_week.py`: source-of-truth planner from GitHub issue body.
- `scripts/run_wave.py`: chunk picker, prompt generator, and state updater.
- `scripts/pr_queue_report.py`: queue status and throughput report.
- `scripts/daily_autorun.py`: unattended daily scheduler/executor over queue state.
- `scripts/install_daily_cron.sh`: idempotent cron installer for daily autorun.

## Workflow

1. Run `python3 scripts/plan_week.py` to fetch issue data and build queue/state.
2. Run `python3 scripts/run_wave.py` to pick the next pending chunk and copy prompt text.
3. Execute chunk work with `fluid-pr-agent`.
4. Update status with `scripts/run_wave.py --chunk-id <id> --set-status <status> [--pr-url URL] [--notes TEXT]`.
5. Run `python3 scripts/pr_queue_report.py` for current progress and throughput estimate.

## Daily unattended runner

`scripts/daily_autorun.py` adds a safety-first unattended mode:

- Optionally refreshes queue/state via `python3 scripts/plan_week.py`.
- Picks active-week pending chunks based on week date ranges in `state.json`.
- Computes daily target as `ceil(pending_in_week / days_left_in_week)` and caps by `--max-chunks-per-day`.
- Falls back to global pending order if active week has no pending chunks.
- Runs selected chunks with `opencode run --agent fluid-pr-agent ...`, logs full output under `automation/logs/YYYY-MM-DD/`, parses outcome tokens, and updates state.

Quick start:

- `python3 scripts/daily_autorun.py --dry-run`
- `bash scripts/install_daily_cron.sh`

Pause switch:

- Pause immediately: `touch automation/PAUSE`
- Optional reason: `printf "maintenance window" > automation/PAUSE`
- Resume: `rm -f automation/PAUSE`
- Override once (manual run): `python3 scripts/daily_autorun.py --ignore-pause`

Safety default:

- No automatic PR open behavior is enabled by default.
- Automatic PR finalization behavior is enabled only by intentional choice:
  - runtime flag: `--auto-open-pr`, or
  - cron install time env: `AUTO_OPEN_PR=1 bash scripts/install_daily_cron.sh`

This means explicit confirmation policy remains the default. It can be bypassed only by intentionally enabling `--auto-open-pr` automation.

## Local queue UI

Use `scripts/ui_server.py` to run a small local UI with Python stdlib only:

- Start server: `python3 scripts/ui_server.py` (default `127.0.0.1:8787`)
- Open `http://127.0.0.1:8787/` to view queue state and run safe actions.

Endpoints exposed by the UI server:

- `GET /api/state`: return `automation/state.json`.
- `POST /api/refresh`: run `python3 scripts/plan_week.py`.
- `POST /api/report`: run `python3 scripts/pr_queue_report.py`.
- `POST /api/dry-run`: run `python3 scripts/daily_autorun.py --dry-run` (optional JSON `date`).
- `POST /api/chunk/update`: update chunk fields (`status`, `pr_url`, `notes`) by `chunk_id`.
- `GET /api/pause`: current pause state.
- `POST /api/pause`: set pause flag and reason by managing `automation/PAUSE`.
- `GET /api/review/state`: review trigger state (`?pr_number=<n>` optional).
- `POST /api/review/run`: run post-review triage for a PR with rounds/max triggers.
- `POST /api/review/reset`: reset trigger counter for a PR.

UI safety notes:

- No endpoint exists to auto-open or auto-create PRs.
- PR actions still require explicit user confirmation.
- API inputs use strict JSON parsing with clear validation errors.
- Post-review triage supports rounds `1-5` and configurable trigger cap (default 5).

## Simple UI Flow

Use this exact order in the local UI:

1. **Refresh queue** with the `Refresh Queue` quick action.
2. **Pick chunk** in the Chunks table using `Select` to prefill the update form.
3. **Run dry-run / execute** with `Daily Dry Run`, then run real execution outside the UI (`fluid-pr-agent`) as needed.
4. **Update status** in `Update Selected Chunk` (status, PR URL, notes) and submit.

## Policy Note

Generated prompts explicitly include:

- Do not open or create any pull request without explicit user confirmation.

This keeps queue automation aligned with current PR approval policy.
